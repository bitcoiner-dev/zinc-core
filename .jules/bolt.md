## 2024-05-26 - Vector Pre-allocation with Unknown Exact Bounds

**Learning:** When pre-allocating `Vec` collections based on iterator lengths, calling `.count()` on the iterator (e.g., `list_unspent().count()`) to get the exact size is an anti-pattern as it consumes the iterator and performs an expensive full iteration, negating the allocation savings. The `Wallet` struct doesn't expose a fast length method for unspent outputs. Furthermore, using a conservative estimate or a loose upper bound for `with_capacity()` must be carefully evaluated to avoid excessive memory allocation, especially when combined with multiple wallets (e.g., vault and payment wallets).

**Action:** In `src/builder.rs`, `collect_utxos` closures use `.list_unspent()` to populate a `known_utxos` HashMap. Since we cannot quickly determine the exact total number of UTXOs beforehand, we should instead iterate once and `.insert()` into a dynamically sizing `HashMap`. The overhead of dynamic resizing is generally less than the cost of a double iteration via `.count()` just to pre-allocate exact capacity. Pre-allocation should only be used when the size is statically known or available via an O(1) `.len()` method. Or if we use values that are already defined in memory like in `src/ordinals/shield.rs` where we can preallocate `Vec` by summing the length of values from an existing map without re-computing the iterators elements.
