## 2024-05-30 - [Panic Risk from unwrap on type conversions]
**Vulnerability:** Found uses of `.unwrap()` on `u32::try_from(vout)` when converting from `usize` in PSBT and transaction output parsing. This can cause the application (and particularly the WASM runtime) to panic and crash if the index exceeds `u32::MAX`, leading to a Denial of Service (DoS).
**Learning:** Type conversions, especially from `usize` to narrower types like `u32` when handling potentially large or external inputs (like PSBT inputs/outputs), are prone to panic if unhandled.
**Prevention:** Avoid using `.unwrap()` or `.expect()` on `TryFrom` conversions in parsing and handling user-provided data. Use safe fallbacks or map errors (e.g. `OrdError::RequestFailed` or returning an error `Result`) instead to fail securely.
## $(date +%Y-%m-%d) - [CRITICAL] Remove unwrap() on ChildNumber derivations
**Vulnerability:** The application used `.unwrap()` when deriving BIP32 child keys via `ChildNumber::from_hardened_idx` and `from_normal_idx`. Since invalid indices (>= 2^31) can panic the application, this creates a Denial of Service (DoS) vulnerability if indices are derived from external state or user inputs.
**Learning:** Even if some indices are hardcoded (e.g., 86, 0) and guaranteed not to panic, applying uniform error propagation (`.map_err(...)?`) across the entire derivation path array is a critical defense-in-depth practice. This ensures that if hardcoded values are ever replaced with dynamic variables, the application remains secure and fails gracefully instead of crashing.
**Prevention:** Always handle the `Result` gracefully when deriving BIP32 child keys, never use `.unwrap()`, and uniformly apply error handling to all elements in a derivation path array.
