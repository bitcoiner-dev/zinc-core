## 2024-05-30 - [Panic Risk from unwrap on type conversions]
**Vulnerability:** Found uses of `.unwrap()` on `u32::try_from(vout)` when converting from `usize` in PSBT and transaction output parsing. This can cause the application (and particularly the WASM runtime) to panic and crash if the index exceeds `u32::MAX`, leading to a Denial of Service (DoS).
**Learning:** Type conversions, especially from `usize` to narrower types like `u32` when handling potentially large or external inputs (like PSBT inputs/outputs), are prone to panic if unhandled.
**Prevention:** Avoid using `.unwrap()` or `.expect()` on `TryFrom` conversions in parsing and handling user-provided data. Use safe fallbacks or map errors (e.g. `OrdError::RequestFailed` or returning an error `Result`) instead to fail securely.
## 2024-05-12 - [Unsafe Type Conversions Replaced]
**Vulnerability:** Found uses of `.unwrap()` on type conversions (e.g., `try_into().unwrap()`, `u8::try_from().unwrap()`) and unsafe integer casts (e.g., `vout as u32`) which could lead to panics and DoS vulnerabilities in Rust/WASM environments when processing untrusted inputs.
**Learning:** External or large datasets processed by methods such as `analyze_psbt_with_scope` and BIP32 derivation functions shouldn't rely on unvalidated type assumptions. Panicking on type conversion can crash the application instead of handling errors gracefully.
**Prevention:** Replaced unsafe conversions with `Result::map_err` to propagate useful error context instead of crashing, and used `unwrap_or` for safe fallbacks on byte length evaluations. Replaced `as u32` with `u32::try_from().map_err()` to prevent integer truncation vulnerabilities.
