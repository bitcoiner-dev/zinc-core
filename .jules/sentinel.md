## 2024-05-30 - [Panic Risk from unwrap on type conversions]
**Vulnerability:** Found uses of `.unwrap()` on `u32::try_from(vout)` when converting from `usize` in PSBT and transaction output parsing. This can cause the application (and particularly the WASM runtime) to panic and crash if the index exceeds `u32::MAX`, leading to a Denial of Service (DoS).
**Learning:** Type conversions, especially from `usize` to narrower types like `u32` when handling potentially large or external inputs (like PSBT inputs/outputs), are prone to panic if unhandled.
**Prevention:** Avoid using `.unwrap()` or `.expect()` on `TryFrom` conversions in parsing and handling user-provided data. Use safe fallbacks or map errors (e.g. `OrdError::RequestFailed` or returning an error `Result`) instead to fail securely.

## 2024-05-10 - Replace unwrap() with error handling in ChildNumber derivation
**Vulnerability:** The code uses `.unwrap()` to extract `bdk_wallet::bitcoin::bip32::ChildNumber` from potentially unsafe or large indices passed into public APIs (e.g., `derive_public_key_internal` and `derive_private_key_internal` using `purpose`, `coin_type`, `account`, `chain`, and `index`). If an unvalidated large index (>= `0x80000000` for `from_normal_idx`) is requested by an external caller or from network data, it causes a panic in `from_hardened_idx` or `from_normal_idx`, leading to Denial of Service (DoS) attacks.
**Learning:** Panics in Rust/WASM are difficult to recover from gracefully and shouldn't be used for input validation logic or boundaries parsing network/external variables. Always use fallible methods like `map_err` instead of `unwrap()`.
**Prevention:** Bound derivation logic properly using `Result` mapping on indices checking inputs at derivation time, preventing `unwrap()` calls on derivation constants or variables passed through public functions.
