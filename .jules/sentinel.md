## 2024-05-30 - [Panic Risk from unwrap on type conversions]
**Vulnerability:** Found uses of `.unwrap()` on `u32::try_from(vout)` when converting from `usize` in PSBT and transaction output parsing. This can cause the application (and particularly the WASM runtime) to panic and crash if the index exceeds `u32::MAX`, leading to a Denial of Service (DoS).
**Learning:** Type conversions, especially from `usize` to narrower types like `u32` when handling potentially large or external inputs (like PSBT inputs/outputs), are prone to panic if unhandled.
**Prevention:** Avoid using `.unwrap()` or `.expect()` on `TryFrom` conversions in parsing and handling user-provided data. Use safe fallbacks or map errors (e.g. `OrdError::RequestFailed` or returning an error `Result`) instead to fail securely.
## 2025-02-13 - [Preventing Panic in BIP-32 Derivation]
**Vulnerability:** Unvalidated derivation index inputs (`purpose`, `account`, `chain`, `index`) when initializing `bdk_wallet::bitcoin::bip32::ChildNumber` used `.unwrap()`, which could lead to panics and DoS attacks if values exceeded `2^31-1`.
**Learning:** Security context demands avoiding panics in public or user-exposed input APIs. Constants or securely bounded parameters can use `.expect("valid constant")`, but variable indices must gracefully return a `Result` via `.map_err()`.
**Prevention:** Use safer type casting and bounds-checking paired with returning informative errors like `.map_err(|e| format!("Invalid child index: {e}"))?` instead of unhandled assumptions that `from_hardened_idx` or `from_normal_idx` will not exceed maximum bounds.
